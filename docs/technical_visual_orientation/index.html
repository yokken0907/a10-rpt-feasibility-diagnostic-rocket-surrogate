1. RPT-specific mission variable

In this repository, the mission variable is not engine thrust or real flight performance. It is a surrogate-level viability target: chamber-pressure state, pressure-oscillation response, thermal load, supply margin, and cumulative-impulse proxy remain diagnostically viable under noise, disturbances, and combined-stress forcing. The question is whether the structured-prior route delays or avoids surrogate-level collapse, not whether a real engine has been designed.

2. Reduced nondimensional surrogate

A10-RPT should be read as a nondimensional reduced-surrogate feasibility archive. It does not define engine geometry, propellant chemistry, injector design, combustion CFD, flight code, or operational procedure. The useful object is the diagnostic map of state channels and feasibility regimes.

RPT-specific surrogate reading

  • The state channels are propulsion-adjacent diagnostics: pressure, oscillation, thermal load, supply margin, cumulative output proxy.
  • The model is nondimensional and reduced; it is not a combustion solver.
  • The value is in stress-channel separation and claim-bounded feasibility screening.
  • Experimental engine validation and safety analysis are outside scope.

3. RPT stress channels

The differentiating feature of A10-RPT is that failure has propulsion-surrogate meanings: pressure oscillation growth, thermal accumulation, supply-margin depletion, cumulative-output loss, or combined-stress collapse.

Why this differs from BMS/STMS

  • BMS failure is SOC / voltage / thermal / plating feasibility.
  • STMS failure is temperature / storage / power / delay feasibility.
  • RPT failure is surrogate pressure-thermal-supply regime collapse.
  • The mission variable is therefore RPT-specific, not a generic A10 placeholder.

4. Regime split: controller-feasible versus architecture-limited

A project-specific strength of A10-RPT is the distinction between feasibility regimes: some stress patterns may be controller-feasible, some may be resource-frontier-feasible, and some may be architecture-limited or unresolved. This is a diagnostic classification, not propulsion readiness.

Diagnostic interpretation

  • Controller-feasible does not mean flight-ready.
  • Resource-frontier-feasible does not mean engine-feasible.
  • Architecture-limited means the reduced model has reached a structural barrier.
  • Unresolved regions should remain unresolved rather than being promoted.

5. Structured-prior diagnostic loop

The A10 prior in this project restricts the surrogate response space so that pressure, oscillation, thermal load, supply margin, and combined-stress regimes can be separated. It is not a claim that A10 replaces propulsion physics.

1. Pressure state
Define the nondimensional pressure-like mission state.
2. Oscillation / growth
Inspect instability-sensitive response.
3. Thermal and supply
Track accumulating load and margin depletion.
4. Structured route
Restrict candidate responses to interpretable regimes.
5. Regime decision
Separate controller-feasible, frontier, and architecture-limited cases.
6. Claim boundary
No engine, flight, safety, or experiment claim.
